## 讨论记录

### 2026/3/31 - 积分并发安全讨论

#### 问题定义

- 500 并发峰值 + 积分不扣错 = 核心难点
- 经典问题：同一用户多设备同时扣减，可能丢失扣减

#### 方案对比

| 方案                       | 性能 | 复杂度 | 适用场景         |
| -------------------------- | ---- | ------ | ---------------- |
| 悲观锁 (SELECT FOR UPDATE) | 中   | 低     | 中低并发         |
| 乐观锁 (版本号)            | 高   | 中     | 读多写少、冲突少 |
| Redis 原子操作             | 极高 | 中     | 高并发热点数据   |
| 分布式锁                   | 低   | 高     | 分布式多实例     |

#### 决策结论

**推荐方案：悲观锁 + 防重**

1. **数据库层**：SELECT FOR UPDATE
   - TypeORM 的 pessimistic_write 锁
   - 锁粒度：用户级别（每个用户只锁自己那一行）

2. **接口层**：防重提交
   - 前端：按钮点击后 disable
   - 后端：同一用户 + 同一商品 + 5秒内 → 拒绝
   - Redis SETNX 做幂等 key

3. **事务边界**

   ```
   BEGIN
     → 锁积分行
     → 检查余额
     → 扣减积分
     → 创建订单
     → 记录日志
   COMMIT
   ```

#### 选择理由

- 500 并发分散到 5000 员工，平均每人同时操作概率低
- MySQL + 悲观锁完全能扛，无需 Redis 做积分
- 避免维护双数据源一致性的复杂度

#### 待产品确认（已补充到 PRD）

- [x] 防重时间窗口：**5秒**
- [x] 秒杀/限量场景：**无**，不需要限购功能
- [x] 错误提示：
  - 积分不足："您的积分不足，当前余额 XX"
  - 库存不足："该商品已被抢光"
  - 重复提交："操作过于频繁，请稍后再试"

**🔒 此议题已关闭，方案确定：悲观锁 + 5秒防重**

### 2026/3/31 - 月度积分发放机制讨论

#### 问题定义

PRD 原文："每个员工有固定积分，每月由系统自动发放"
需要明确：发放时间、金额规则、历史积分处理、入离职场景、手动调整

#### 方案对比

| 模式       | 描述               | 优点             | 缺点                 |
| ---------- | ------------------ | ---------------- | -------------------- |
| 清零模式   | 月底清零，下月重发 | 简单，无历史负担 | 员工可能月底扎堆兑换 |
| 累积模式   | 无上限累积         | 灵活             | 企业财务负债大       |
| 有效期模式 | 每批积分独立有效期 | 平衡             | 实现复杂             |
| 年度额度   | 年初一次性发放     | 员工自主规划     | 年底扎堆             |

#### 决策结论 ✅

选择 **清零模式**（最简单），具体规则：

```
┌─────────────────────────────────────────────────────────────┐
│                   月度积分发放规则                           │
├─────────────────────────────────────────────────────────────┤
│  发放时间：每月 1 号 0:00 自动发放                          │
│  历史积分：月底清零，不累积                                  │
│  发放金额：按员工级别不同金额（后台配置）                    │
│  月中入职：等下月再发放（当月不发）                          │
│  月中离职：积分立即清零，不可再用                            │
│  手动调整：HR 可手动调整，无需审批                          │
└─────────────────────────────────────────────────────────────┘
```

#### 技术实现要点

1. **定时任务**：每月 1 号 0:00 触发发放任务
2. **表设计简化**：只需 `balance` 字段，无需积分批次表
3. **清零逻辑**：发放前先将所有在职员工余额清零
4. **级别配置**：需要 `employee_level` + `level_points_config` 表
5. **操作日志**：所有调整都记录（自动发放 / 手动调整 / 离职清零）

#### 发放流程

```
每月1号 0:00
     │
     ▼
┌──────────────┐
│ 获取所有在职 │
│  员工列表    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 清零所有余额 │ ──▶ 记录日志：monthly_reset
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 按级别发放   │ ──▶ 记录日志：monthly_grant
```

### 2026/3/31 - 库存预扣 vs 实扣讨论

#### 问题定义

兑换涉及两个资源扣减（积分 + 库存），必须保证原子性：

- 要么都成功
- 要么都失败
- 不能出现"积分扣了但库存没了"或"库存扣了但积分不够"

#### 方案对比

| 方案      | 复杂度 | 一致性 | 性能 | 适用场景       |
| --------- | ------ | ------ | ---- | -------------- |
| 本地事务  | 低 ⭐  | 强 ⭐  | 中   | 单体/同库      |
| 预扣+确认 | 中     | 最终   | 高   | 需等待支付场景 |
| Saga/TCC  | 高     | 最终   | 中   | 微服务/跨库    |

#### 决策结论 ✅

选择 **本地事务** 方案（最简单有效）

```
┌─────────────────────────────────────────────────────────────┐
│                   兑换事务流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  BEGIN TRANSACTION                                          │
│  │                                                          │
│  ├──▶ 1. 锁定积分行 (SELECT FOR UPDATE)                    │
│  │       检查余额 ≥ 所需积分                                │
│  │                                                          │
│  ├──▶ 2. 锁定库存行 (SELECT FOR UPDATE)                    │
│  │       检查库存 ≥ 购买数量                                │
│  │                                                          │
│  ├──▶ 3. 扣减积分                                          │
│  │                                                          │
│  ├──▶ 4. 扣减库存                                          │
│  │                                                          │
│  ├──▶ 5. 创建订单                                          │
│  │                                                          │
│  ├──▶ 6. 记录日志                                          │
│  │                                                          │
│  COMMIT / ROLLBACK                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 选择理由

- 单体架构，积分和库存在同一数据库
- 福利兑换是即时扣减，不需要预扣模式
- 本地事务由 MySQL 保证 ACID，最可靠
- 实现简单，TypeORM 原生支持

#### 关键实现点

1. **锁顺序固定**：先积分后库存，避免死锁
2. **悲观锁**：SELECT FOR UPDATE
3. **事务隔离**：默认 READ COMMITTED 即可
4. **日志记录**：积分变动和库存变动都要记录

**🔒 此议题已关闭，方案确定：本地事务**
