## 2. 命名规范

### 2.1 文件命名

| 类型           | 规范                 | 示例                                       |
| -------------- | -------------------- | ------------------------------------------ |
| 后端模块文件   | kebab-case（短横线） | `user.controller.ts`, `create-user.dto.ts` |
| 前端组件文件   | PascalCase（大驼峰） | `UserList.tsx`, `OrderDetail.tsx`          |
| 前端工具/hooks | camelCase（小驼峰）  | `useAuth.ts`, `formatDate.ts`              |

### 2.2 变量/函数命名

| 类型     | 规范                 | 示例                              |
| -------- | -------------------- | --------------------------------- |
| 变量     | camelCase            | `userId`, `orderList`             |
| 布尔变量 | is/has/can 前缀      | `isActive`, `hasPermission`       |
| 常量     | UPPER_SNAKE_CASE     | `MAX_RETRY_COUNT`                 |
| 函数     | camelCase + 动词开头 | `getUserById()`, `handleSubmit()` |

### 2.3 类/接口命名

| 类型   | 规范                      | 示例                              |
| ------ | ------------------------- | --------------------------------- |
| 类     | PascalCase                | `UserService`                     |
| 接口   | PascalCase（不加 I 前缀） | `User`, `Order`                   |
| DTO    | XxxDto                    | `CreateUserDto`, `UpdateOrderDto` |
| VO     | XxxVo                     | `UserListVo`, `OrderDetailVo`     |
| Entity | 单数名词                  | `User`, `Order`, `Product`        |

## 3. API 设计规范

### 3.1 URL 设计

- 资源用名词复数：`/api/users`, `/api/orders`
- 层级关系：`/api/users/{id}/orders`
- 全小写 + 短横线：`/api/point-logs`

### 3.2 HTTP 方法

| 方法   | 路径             | 说明     |
| ------ | ---------------- | -------- |
| GET    | `/api/users`     | 获取列表 |
| GET    | `/api/users/:id` | 获取详情 |
| POST   | `/api/users`     | 创建     |
| PUT    | `/api/users/:id` | 全量更新 |
| PATCH  | `/api/users/:id` | 部分更新 |
| DELETE | `/api/users/:id` | 删除     |

### 3.3 非 CRUD 操作

```
POST /api/orders/:id/ship      # 发货
POST /api/orders/:id/cancel    # 取消
POST /api/auth/login           # 登录
```

### 3.4 统一响应格式

**成功响应：**

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

**分页响应：**

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [...],
    "total": 100,
    "page": 1,
    "pageSize": 10
  }
}
```

**失败响应：**

```json
{
  "code": 10001,
  "message": "用户不存在",
  "data": null
}
```

### 3.5 错误码规范

| 范围        | 模块                           |
| ----------- | ------------------------------ |
| 0           | 成功                           |
| 10000-19999 | 通用错误（参数错误、未授权等） |
| 20000-29999 | 用户/员工模块                  |
| 30000-39999 | 商品模块                       |
| 40000-49999 | 订单模块                       |
| 50000-59999 | 积分模块                       |
| 60000-69999 | 系统模块                       |

**常用错误码：**
| 错误码 | 说明 |
|--------|------|
| 10001 | 参数校验失败 |
| 10002 | 未登录 |
| 10003 | 无权限 |
| 10004 | 资源不存在 |
| 10005 | 操作过于频繁 |
| 30001 | 库存不足 |
| 50001 | 积分不足 |

## 4. 数据库规范

### 4.1 表名规范

- 小写 + 下划线
- 使用前缀区分模块：

| 前缀  | 模块                               |
| ----- | ---------------------------------- |
| sys\_ | 系统模块（用户、角色、权限、日志） |
| prd\_ | 商品模块                           |
| ord\_ | 订单模块                           |
| pts\_ | 积分模块                           |

**示例：** `sys_user`, `prd_product`, `ord_order`, `sys_user_role`

### 4.2 字段规范

- 小写 + 下划线：`user_name`, `create_time`
- 主键：`id`（BIGINT 自增）
- 外键：`xxx_id`（如 `user_id`）
- 布尔值：`is_xxx`（如 `is_active`）

### 4.3 必备字段

| 字段        | 类型       | 说明                  |
| ----------- | ---------- | --------------------- |
| id          | BIGINT     | 主键自增              |
| create_time | DATETIME   | 创建时间              |
| update_time | DATETIME   | 更新时间（ON UPDATE） |
| create_by   | BIGINT     | 创建人ID（可选）      |
| update_by   | BIGINT     | 更新人ID（可选）      |
| is_deleted  | TINYINT(1) | 软删除标记（0否1是）  |

### 4.4 索引命名

| 类型     | 命名        |
| -------- | ----------- |
| 主键索引 | pk_xxx      |
| 唯一索引 | uk_xxx      |
| 普通索引 | idx_xxx     |
| 组合索引 | idx_xxx_yyy |

## 5. Git 规范

### 5.1 分支策略

| 分支        | 说明       |
| ----------- | ---------- |
| main        | 生产环境   |
| develop     | 开发主分支 |
| feature/xxx | 功能分支   |
| fix/xxx     | 修复分支   |
| hotfix/xxx  | 紧急修复   |

### 5.2 Commit 规范

**格式：**

```
<type>(<scope>): <subject>
```

**type 类型：**
| type | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复 bug |
| docs | 文档更新 |
| style | 代码格式（不影响逻辑） |
| refactor | 重构 |
| perf | 性能优化 |
| test | 测试相关 |
| chore | 构建/工具变动 |
| revert | 回滚 |

**scope 范围（可选）：** server, admin, shared, docker, docs

**示例：**

```
feat(server): 添加用户登录接口
fix(admin): 修复订单列表分页问题
docs: 更新 README
```

## 6. 代码分层规范

### 6.1 后端分层

```
Controller → Service → Repository → Entity
```

| 层         | 职责                                      |
| ---------- | ----------------------------------------- |
| Controller | 接收请求、参数校验（DTO）、返回响应（VO） |
| Service    | 业务逻辑、事务管理                        |
| Repository | 数据访问、复杂查询                        |
| Entity     | 数据库实体定义                            |

### 6.2 禁止事项（红线）

- ❌ Controller 里写业务逻辑
- ❌ Service 里写原生 SQL
- ❌ 循环查库（N+1 问题）
- ❌ 事务范围过大
- ❌ 硬编码配置
- ❌ 不做参数校验
- ❌ 密码明文存储
- ❌ 接口不做权限校验

## 7. 安全规范

- 密码使用 bcrypt 加密存储
- JWT Token 设置合理过期时间
- 敏感接口添加防重（幂等）
- 日志脱敏（不记录密码等敏感信息）
- 使用参数化查询，防止 SQL 注入
- 输入校验，防止 XSS
